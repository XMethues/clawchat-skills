# Liveware Server Prompt Architecture

> 这份参考文档描述了 `liveware/server.py` 中 `_build_card_prompt()` 构建的 LLM prompt 结构，以及输出格式要求的最佳实践。

## Prompt Structure

`server.py` 的 `_build_card_prompt()` 构建一个发送给 Hermes API Server 的 prompt，结构如下：

```
## 任务
[描述角色：tarot-arcana liveware 网页提交的解读请求]
[说明：必须使用 tarot-arcana 技能完成分析，不是独立引擎]

## 必须使用的技能
1. skill_view(name='tarot-arcana')
2. skill_view(name='tarot-arcana', file_path='references/interpretation-rules.md')
3. skill_view(name='tarot-arcana', file_path='references/spreads.md')
4. [说明：只解读已提交的牌，不要重新抽牌]

## 用户问题
[用户填写的问题原文]

## 已提交牌阵（N张牌）
[每张牌的详细数据：位置、名称、逆位状态、关键词等]

## 输出要求
[见下文]
```

## 输出要求的最佳实践

### 正确版本（经过验证）

```
## 输出要求

请提供完整、有结构、温和且有行动建议的塔罗解读。解读内容应当包括：

1. **单牌解读**——对每张牌的含义和位置意义的解读
2. **综合分析**——牌与牌之间的联系和整体故事
3. **建议**——基于牌面给出的行动指引或方向

解读中**不要**出现以下内容：

- 不要单独列出"你问的是什么"或描述用户问题
- 不要说明"牌阵"类型或牌阵说明
- 不要出现解读过程的元说明（如"我将从过去开始分析..."）
- 不要使用 `---` 或 `***` 等分割线——解读正文应当连续，无需视觉分隔

把位置意义融入每张牌的解读中，自然连贯。
```

### 关键原则

1. **显式列出要包含的内容**——只禁止"不要"是不够的。要明确告诉 LLM 应该输出哪些部分（单牌解读、综合分析、建议），否则模型会过度收敛，连有用的内容也砍掉。

2. **精确界定禁止范围**——禁止的应该是元信息（重复问题、解释牌阵类型、分析过程描述），而不是内容本身（建议部分）。

3. **"位置意义"不要单独禁掉**——之前尝试过"不要解释位置意义"，导致 LLM 完全跳过位置相关的解读。正确做法是要求"把位置意义融入每张牌的解读中"。

4. **避免过于宽泛的禁止**——"不要出现任何元说明或结构注释"太模糊，模型会把"建议"也当作元说明砍掉。用具体例子说明什么是元说明（如"我将从过去开始分析…"）。

### 之前踩过的坑

| 版本 | 问题 | 结果 |
|------|------|------|
| 太宽泛 | `不要出现任何元说明或结构注释` | 建议、综合分析等有用内容全被砍掉 |
| 禁止位置意义 | `不要解释"位置意义"` | LLM 完全跳过位置解读 |
| 没禁止分割线 | 输出中出现 `---` / `***` 等视觉分隔 | 用户认为分割线多余，要求去掉 |
| 正确版本 | 显式列出三块内容 + 精准禁止元信息 + 禁止分割线 | 结构完整、有建议、无元废话、无分割线 |

## 测试方式

提交一个解读请求后，检查返回的解读文本：
- ✅ 有单牌解读（每张牌的含义）
- ✅ 有综合分析（牌与牌之间的联系）
- ✅ 有建议（行动指引）
- ❌ 没有"你问的是…"等重复问题
- ❌ 没有"这是三张牌牌阵"等牌阵说明
- ❌ 没有"我将从过去开始分析"等过程描述
